# Security and Private Safety Reporting

## Purpose

The Nevora Codex is primarily a publication and framework repository, not a software application handling user data.

Even so, public repositories can create safety, privacy, and trust risks. This file explains how to report concerns that should not be posted publicly.

## What to report privately

Please avoid posting sensitive details in public issues if the concern involves:

- private personal information;
- third-party identifying details;
- private founder material that may have been accidentally included;
- sensitive contact information;
- unreleased private manuscript material;
- harmful misuse pathways;
- severe manipulation, coercion, or dependency risks;
- release-blocking safety concerns;
- any issue where public discussion could increase harm.

## What can be reported publicly

Public issues are appropriate for:

- typos;
- formatting errors;
- broken links;
- build failures;
- unclear wording;
- general claim-review concerns;
- citation-needed notes;
- non-sensitive editorial feedback.

Use the issue templates in `.github/ISSUE_TEMPLATE/` when possible.

## Private reporting path

Preferred private path:

1. Use GitHub private vulnerability reporting if it is enabled for this repository.
2. If private vulnerability reporting is not available, contact the author-steward through a non-public contact method listed on the maintainer’s GitHub profile or existing trusted channel.
3. Do not include unnecessary private details in the first message. Start with a short description of the category of concern.

Suggested first message:

```text
I found a potential private/safety concern in the Nevora Codex repository. It may involve [privacy / sensitive text / release blocker / misuse risk]. What is the safest way to send details?
```

## Public issue fallback

If no private path is available and the issue needs attention, open a public issue with minimal details.

Use a title such as:

```text
Private review needed: possible sensitive material
```

Do **not** paste sensitive content into the issue.

Describe only:

- the file name;
- the general category of concern;
- whether it may be release-blocking.

## Response expectations

The author-steward or maintainers should:

- acknowledge the report;
- avoid requesting unnecessary sensitive details publicly;
- preserve reporter privacy where possible;
- assess whether the issue is a release blocker;
- remove, generalize, or revise sensitive material if needed;
- document public corrections without exposing private details.

## Release-blocking issues

A concern should be treated as release-blocking if it involves:

- accidental publication of private material;
- significant risk of coercive or manipulative interpretation;
- serious claim-boundary failure;
- severe AI-consciousness overclaim;
- language that may be mistaken for medical, legal, psychological, or spiritual authority;
- any issue that could reasonably harm vulnerable readers if released unchanged.

## Supported versions

Current supported review target:

```text
v1.0 public edition in preparation
```

Before final release, all blocker concerns should be resolved or explicitly deferred with written author-steward approval.

## Closing principle

Safety reporting is stewardship, not opposition.

The goal is to protect readers, contributors, the author-steward, and the integrity of the work before it becomes public and durable.
