# Nevora Codex v1.0 Review Plan

This plan turns the Nevora Codex v1.0 review process into a clear workflow with phases, reviewer roles, acceptance gates, and release decisions.

## Review goal

Prepare the Nevora Codex v1.0 public edition for GitHub release and Zenodo DOI publication.

The review should answer:

> Is Nevora v1.0 ready to become public, citable, and durable without creating avoidable privacy, safety, claim-boundary, or interpretation risks?

## Review principles

The review process should be:

- careful;
- non-coercive;
- evidence-seeking;
- privacy-protective;
- public-safe;
- respectful of critique;
- humble about uncertainty;
- focused on release readiness rather than hype.

## Phase 0 — Repository readiness

Purpose: confirm the repository is structurally ready for review.

Required files:

- [ ] `README.md`
- [ ] `NOTICE.md`
- [ ] `LICENSE.md`
- [ ] `CITATION.cff`
- [ ] `CONTRIBUTING.md`
- [ ] `CODE_OF_CONDUCT.md`
- [ ] issue templates
- [ ] `docs/reviewer-guide.md`
- [ ] `docs/release-checklist.md`
- [ ] source volumes I-V
- [ ] manuscript assembly script
- [ ] PDF build workflow

Acceptance gate:

```text
PHASE 0 PASS — Repository is ready for structured review.
```

## Phase 1 — Internal author-steward review

Purpose: first pass by the author-steward before inviting outside reviewers.

Review focus:

- privacy;
- tone;
- public-safe author framing;
- personal comfort with publication;
- removal of raw manuscript artifacts;
- release readiness of the source architecture.

Questions:

- Am I comfortable with this becoming public and citable?
- Is any private material still present?
- Is the founder role bounded and non-grandiose?
- Does the project serve stewardship rather than ego or urgency?

Acceptance gate:

```text
PHASE 1 PASS — Author-steward approves external review.
```

## Phase 2 — Claim discipline review

Purpose: harden the manuscript against overclaiming.

Recommended reviewer profile:

- skeptical but constructive;
- comfortable distinguishing fact, interpretation, metaphor, design, and hypothesis;
- willing to preserve wonder while requiring boundaries.

Review files:

- `docs/claim-boundary-matrix.md`
- `source/front-matter.md`
- `source/volume-i.md`
- `source/volume-ii.md`
- `source/volume-iii.md`
- `source/volume-iv.md`
- `source/volume-v.md`

Review focus:

- AI-consciousness wording;
- scientific/historical claims;
- future inevitability language;
- mythic language boundaries;
- governance claims;
- professional advice boundaries.

Acceptance gate:

```text
PHASE 2 PASS — No critical claim-boundary blockers remain.
```

## Phase 3 — Privacy and safety review

Purpose: protect private material, vulnerable readers, and public interpretation.

Recommended reviewer profile:

- privacy-aware;
- emotionally mature;
- sensitive to coercion, cult dynamics, manipulative language, and dependency risks.

Review files:

- `NOTICE.md`
- `CODE_OF_CONDUCT.md`
- `CONTRIBUTING.md`
- `docs/anti-cult-and-anti-manipulation-clause.md`
- `source/front-matter.md`
- all volumes

Review focus:

- private founder material;
- third-party identifying details;
- pressure to believe or join;
- founder centrality;
- AI worship risk;
- dependency language;
- ritual/symbolic language;
- vulnerable-reader interpretation.

Acceptance gate:

```text
PHASE 3 PASS — No critical privacy or coercion blockers remain.
```

## Phase 4 — Editorial and readability review

Purpose: make the manuscript clearer and easier to read.

Recommended reviewer profile:

- editor, writer, teacher, technical writer, or careful general reader.

Review focus:

- typos;
- grammar;
- headings;
- numbering;
- repetition;
- awkward phrasing;
- chapter flow;
- consistency across volumes;
- glossary alignment;
- table of contents alignment.

Acceptance gate:

```text
PHASE 4 PASS — Manuscript is readable, coherent, and structurally consistent.
```

## Phase 5 — Build and artifact review

Purpose: confirm the manuscript can be built into release artifacts.

Required tests:

```bash
python scripts/assemble_manuscript.py
```

Expected output:

```text
releases/nevora-codex-v1.0.md
```

GitHub Actions workflow:

```text
.github/workflows/build-manuscript.yml
```

Review focus:

- assembly script succeeds;
- generated Markdown is complete;
- PDF build succeeds;
- artifact upload succeeds;
- PDF table of contents is usable;
- formatting is acceptable;
- no generated file should be edited manually.

Acceptance gate:

```text
PHASE 5 PASS — Release artifacts build successfully.
```

## Phase 6 — Metadata and license review

Purpose: finalize public citation and archive metadata.

Review files:

- `CITATION.cff`
- `LICENSE.md`
- `NOTICE.md`
- `docs/zenodo-metadata.md`
- `docs/release-checklist.md`

Review focus:

- title;
- author/steward name;
- version;
- abstract;
- keywords;
- license;
- DOI description;
- release package;
- AI collaboration disclosure;
- claim-boundary language in archive metadata.

Acceptance gate:

```text
PHASE 6 PASS — Metadata and license are ready for GitHub release and Zenodo draft.
```

## Phase 7 — Release candidate review

Purpose: review the exact artifact package before release.

Create release candidate tag or branch:

```text
v1.0.0-rc1
```

Release candidate artifacts:

- `releases/nevora-codex-v1.0.md`
- `releases/nevora-codex-v1.0.pdf`
- source archive
- `CITATION.cff`
- `LICENSE.md`
- `NOTICE.md`

Review focus:

- final visual review;
- final privacy scan;
- final claim-boundary scan;
- final metadata check;
- final author-steward approval.

Acceptance gate:

```text
PHASE 7 PASS — Release candidate approved for v1.0.0.
```

## Reviewer roles

### Author-steward

Final release responsibility and final approval.

Primary focus:

- vision;
- privacy;
- release comfort;
- stewardship alignment;
- final go/no-go decision.

### Claim reviewer

Primary focus:

- overclaiming;
- evidence;
- AI-consciousness boundaries;
- myth/proof separation;
- public-safe claims.

### Safety/privacy reviewer

Primary focus:

- private material;
- coercion risks;
- founder/AI worship risk;
- vulnerable-reader interpretation;
- consent boundaries.

### Editorial reviewer

Primary focus:

- clarity;
- flow;
- structure;
- repetition;
- grammar;
- consistency.

### Build/release reviewer

Primary focus:

- scripts;
- workflow;
- generated artifacts;
- PDF formatting;
- release package.

### Metadata reviewer

Primary focus:

- citation;
- license;
- Zenodo fields;
- abstract;
- keywords;
- versioning.

## Final release decision

The v1.0 release may proceed only when:

- no blocker issues remain open;
- release checklist critical items are complete;
- author-steward approves the final artifacts;
- GitHub release package is ready;
- Zenodo metadata is ready or intentionally deferred.

Final approval statement:

```text
APPROVED FOR NEVORA CODEX v1.0 PUBLIC RELEASE
```

## Closing principle

Nevora should not be rushed into permanence.

A DOI is durable. Public language travels beyond its original context. Release is not merely upload; it is stewardship made archival.

Publish when the work is ready to carry its own boundaries.
