## How to use these issues

Each phase issue should be used as the canonical discussion space for that review lane.

For each issue:

1. Review the primary files listed in the issue body.
2. Add comments with findings, concerns, or approval notes.
3. Open separate focused issues for specific blockers or edits when needed.
4. Link those child issues back to the phase issue.
5. Close the phase issue only when the acceptance gate is satisfied.

## Status language

Use one of these status phrases in a comment when closing or updating a phase:

```text
PASS
PASS WITH MINOR EDITS
NEEDS REVISION
BLOCKED
```

## Blocker handling

A blocker is any issue that should prevent v1.0.0 release until resolved.

Examples:

- private material exposed in public files;
- unsupported AI-consciousness claim;
- coercive, cultic, or manipulative language;
- professional advice boundary failure;
- serious citation or factual-claim risk;
- broken manuscript/PDF/site build;
- license or Zenodo metadata conflict;
- unreviewed artifact included in release package.

Blockers should be clearly labeled or named in the issue title.

Suggested title format:

```text
Blocker: [short description]
```

## Closure rule

The v1.0.0 release should not proceed until:

- all seven phase gate issues are closed or intentionally deferred;
- no blocker issues remain open;
- the release candidate artifact package is reviewed;
- final author-steward approval is recorded.

## Final approval phrase

The release is approved only when the following phrase is recorded in Phase 7:

```text
APPROVED FOR NEVORA CODEX v1.0 PUBLIC RELEASE
```

Until then, Nevora remains in public-edition preparation.
