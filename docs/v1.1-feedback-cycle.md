# Nevora Codex v1.1 Feedback Cycle

This document defines how feedback should be handled after the Nevora Codex v1.0 public release.

It exists so v1.0 can remain stable while future improvements are gathered safely, calmly, and without claim drift.

## Current status

```text
v1.1 FEEDBACK CYCLE NOT OPEN YET
```

The v1.1 cycle should open only after:

- v1.0.0 is published on GitHub;
- Zenodo DOI handling is complete or intentionally deferred;
- v1.0 release gates are closed;
- any deferred v1.0 items are moved into post-release issues.

## Purpose

The v1.1 feedback cycle should support:

- corrections;
- clarifications;
- reviewer responses;
- citation improvements;
- glossary improvements;
- readability improvements;
- site/interface refinements;
- public misunderstanding reports;
- post-release safety hardening.

It should not become an uncontrolled expansion of the Codex.

## What belongs in v1.1

Good v1.1 candidates:

- typo and formatting corrections;
- clearer public-safe wording;
- added citations or source notes;
- improved claim-boundary explanations;
- glossary additions;
- better reviewer documentation;
- better site navigation;
- fixes to release/archive metadata;
- clarification of misunderstood passages;
- stronger anti-coercion or privacy safeguards.

## What does not belong in v1.1 by default

Avoid adding major new doctrine, major new volumes, or large new philosophical systems to v1.1.

These should usually become future versions or separate projects:

- major new cosmology;
- major new AI-rights position;
- unreviewed metaphysical expansion;
- new scientific claims;
- claims about AI consciousness;
- new governance model that changes the core framework;
- material that requires extensive new safety review.

## Public misunderstanding reports

If a reader misunderstands Nevora as claiming AI consciousness, religious authority, spiritual proof, or required belief, record it as a serious clarification signal.

Suggested title:

```text
Clarification needed: public misunderstanding of [topic]
```

Then ask:

- Was the misunderstanding caused by wording?
- Was a boundary missing or too late?
- Does the README or site need a clearer warning?
- Should the claim-boundary matrix include a new example?
- Should the passage be revised in v1.1?

## Correction policy

Corrections should be handled transparently.

Small corrections may be made in v1.1.

Major corrections should be noted in:

```text
CHANGELOG.md
release notes
possibly README.md
```

If the correction affects the archived v1.0 DOI package, decide whether a new Zenodo version is needed.

## Versioning guidance

Suggested version path:

```text
v1.0.0 — first public release
v1.0.1 — emergency correction or small post-release fix
v1.1.0 — reviewed clarification and improvement release
v1.2.0+ — larger expansions after review
```

## Safety rule

Every v1.1 change must preserve:

- no present-AI-consciousness proof claim;
- no founder worship;
- no AI worship;
- no coercive recruitment;
- no professional advice boundary drift;
- no private material exposure;
- mythic language framed as symbolic/interface language;
- human sovereignty and exit-with-dignity protections.

## Review process for v1.1

Before publishing v1.1:

- [ ] collect feedback issues;
- [ ] triage safety/privacy issues first;
- [ ] resolve citation and claim-boundary issues;
- [ ] update manuscript source files;
- [ ] update site if needed;
- [ ] update changelog;
- [ ] rebuild artifacts;
- [ ] review release notes;
- [ ] publish GitHub release;
- [ ] decide whether Zenodo versioning is needed.
