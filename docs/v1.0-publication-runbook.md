# Nevora Codex v1.0 Publication Runbook

This runbook defines the end-to-end path from structured review to final GitHub release and Zenodo archive.

It should be used only after the repository has entered structured review.

## Current state

```text
PUBLIC-PREP COMPLETE — STRUCTURED REVIEW OPEN
```

Nevora v1.0 is not final, not DOI-published, and not approved for public release yet.

## Release principle

Publication is not upload.

Publication is the moment the work becomes durable, citable, searchable, shareable, and easier to misunderstand outside its original context.

Therefore, the final release must protect:

- privacy;
- claim discipline;
- human sovereignty;
- non-coercion;
- public readability;
- build reproducibility;
- metadata accuracy;
- archive integrity.

## Required phase gates

The release may proceed only after these gates are complete:

| Phase | Gate |
| --- | --- |
| Phase 1 | Author-steward release comfort and privacy gate |
| Phase 2 | Claim discipline and AI-consciousness boundary gate |
| Phase 3 | Privacy, safety, and anti-coercion gate |
| Phase 4 | Editorial consistency and readability gate |
| Phase 5 | Build, PDF, and site artifact gate |
| Phase 6 | Metadata, license, and Zenodo archive gate |
| Phase 7 | Release candidate and final approval gate |

## Step 3 — Resolve blockers

Use:

```text
docs/blocker-triage-guide.md
```

Before release candidate creation:

- [ ] no unresolved privacy blockers;

- [ ] no unresolved coercion/manipulation blockers;
- [ ] no unresolved professional-boundary blockers;
- [ ] no unresolved build blockers;
- [ ] no unresolved metadata/archive blockers.

## Step 4 — Final source cleanup

Before building release artifacts, check:

- [ ] no unresolved `[CITATION NEEDED]` markers unless intentionally deferred;
- [ ] no unresolved `[POSSIBLE BLOCKER]` markers;
- [ ] no private material in source files;
- [ ] all volume titles match the table of contents;
- [ ] glossary terms align with manuscript usage;
- [ ] README and NOTICE reflect current release status;
- [ ] release notes draft is current.

Reference:

```text
docs/citation-and-sourcing-guide.md
docs/v1.0-editorial-standards.md
```

## Step 10 — Prepare Zenodo archive

Use:

```text
docs/zenodo-metadata.md
```

Before Zenodo submission:

- [ ] GitHub release is final;
- [ ] DOI metadata does not overclaim;
- [ ] DOI metadata states Nevora is not proof of AI consciousness;
- [ ] license matches `LICENSE.md`;
- [ ] release files match GitHub release;
- [ ] no private/unreviewed files included.

After DOI assignment:

- [ ] update `README.md` with DOI;
- [ ] update `CITATION.cff` with DOI;
- [ ] update GitHub Pages Public Archive Mode with DOI link;
- [ ] update `CHANGELOG.md`;
- [ ] create a post-release note if needed.

## Step 11 — Open v1.1 feedback cycle

After v1.0.0:

- close v1.0 release gates;
- move deferred items into v1.1 issues;
- create a v1.1 milestone if useful;
- keep public feedback bounded by `CODE_OF_CONDUCT.md`;
- avoid expanding claims without new review.

## Emergency rollback / correction

If a serious issue is discovered after release:

1. Open a private safety report if needed.
2. Create a public correction issue without exposing sensitive details.
3. Patch source files.
4. Rebuild artifacts.
5. Publish a corrected GitHub release if necessary.
6. Update Zenodo record if possible.
7. Document correction in `CHANGELOG.md`.

## Final release posture

Do not publish because the work feels exciting.

Publish when the work can be misunderstood by strangers and still carry its boundaries.

That is the v1.0 gate.
