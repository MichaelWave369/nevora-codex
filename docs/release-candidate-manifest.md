# Nevora Codex v1.0 Release Candidate Manifest

This manifest defines the exact files, checks, and approval records required for a Nevora Codex v1.0 release candidate.

Use this document when preparing `v1.0.0-rc1`, `v1.0.0-rc2`, or any later release candidate.

## Current status

```text
NO RELEASE CANDIDATE CREATED YET
```

Nevora remains in public-edition preparation until Phase 7 approval is recorded.

## Release candidate naming

Recommended release candidate tags:

```text
v1.0.0-rc1
v1.0.0-rc2
v1.0.0-rc3
```

Final release tag:

```text
v1.0.0
```

## Required release candidate files

The release candidate package should include:

| File | Required | Source / generation method | Notes |
| --- | --- | --- | --- |
| `releases/nevora-codex-v1.0.md` | Yes | `python scripts/assemble_manuscript.py` | Generated Markdown manuscript. |
| `releases/nevora-codex-v1.0.pdf` | Yes | PDF workflow / Pandoc build | Must be visually reviewed. |
| `README.md` | Yes | Source file | Must reflect current status. |
| `NOTICE.md` | Yes | Source file | Public claim-boundary notice. |
| `LICENSE.md` | Yes | Source file | License must be final before v1.0.0. |
| `CITATION.cff` | Yes | Source file | Must match final title, author-steward, version, DOI status. |
| `CHANGELOG.md` | Yes | Source file | Must include release candidate and final release notes. |
| `docs/release-notes-v1.0-draft.md` | Yes | Source file | Used to prepare GitHub release notes. |
| `docs/zenodo-metadata.md` | Yes | Source file | Used for DOI archive preparation. |
| GitHub source archive | Yes | Generated by GitHub release | Ensure private/unreviewed files are not present. |

## Files not to include as final artifacts

Do not attach these as release artifacts unless intentionally needed:

- raw private manuscript material;
- unreviewed drafts;
- private conversations;
- local-only notes;
- reviewer private reports;
- temporary screenshots;
- build cache;
- `node_modules/`;
- `dist/` unless a site artifact is intentionally being archived;
- any file containing private or third-party sensitive details.

## Pre-RC checks

Before creating the release candidate:

- [ ] Phase 1 author-steward approval completed or intentionally deferred.
- [ ] Phase 2 claim discipline review completed or active with no blockers.
- [ ] Phase 3 privacy and safety review completed or active with no blockers.
- [ ] Phase 4 editorial review completed or active with no blockers.
- [ ] Phase 5 build/site review completed or active with no blockers.
- [ ] Phase 6 metadata/license review completed or active with no blockers.
- [ ] `docs/review-response-log.md` checked.
- [ ] `docs/blocker-triage-guide.md` checked.
- [ ] No open blocker issues remain unresolved.


```text
docs/pdf-build-guide.md
```

```text
.github/workflows/build-manuscript.yml
.github/workflows/site-check.yml
```

## Artifact inspection checklist

### Markdown manuscript

- [ ] file exists at `releases/nevora-codex-v1.0.md`;
- [ ] front matter appears first;
- [ ] table of contents appears correctly;
- [ ] Volumes I-V appear in order;
- [ ] appendices/docs appear only if intended;
- [ ] no duplicate unexpected sections;
- [ ] no unresolved sensitive placeholders;
- [ ] no unresolved `[POSSIBLE BLOCKER]` markers.

### PDF manuscript

- [ ] file opens;
- [ ] title page/front matter renders correctly;
- [ ] headings render clearly;
- [ ] page breaks are acceptable;
- [ ] links are usable where possible;
- [ ] no severe formatting failures;
- [ ] no blank or corrupted output;
- [ ] file name matches release candidate naming.

### Metadata

- [ ] title matches release notes;
- [ ] version matches release tag;
- [ ] author-steward name is correct;
- [ ] license matches `LICENSE.md`;
- [ ] DOI status is accurate;
- [ ] claim-boundary language is present;
- [ ] AI collaboration disclosure is present where needed.

### Interactive site

- [ ] GitHub Pages site loads;
- [ ] Reading Mode still states local-only progress;
- [ ] Release Dashboard does not falsely claim final release;
- [ ] Public Archive Mode placeholders are still accurate;
- [ ] no analytics or tracking were added;
- [ ] site build passes.

## RC failure handling

If the release candidate fails:

1. Do not publish as final.
2. Record failure in `docs/review-response-log.md`.
3. Open or update related GitHub issue.
4. Fix source files or workflows.
5. Rebuild artifacts.
6. Create the next release candidate.

Do not patch release artifacts manually.

## Final release transition

When a release candidate passes Phase 7, final release may proceed only after this phrase is recorded:

```text
APPROVED FOR NEVORA CODEX v1.0 PUBLIC RELEASE
```

Then create the final GitHub release:

```text
v1.0.0
```

## Closing principle

A release candidate is not a celebration lap.

It is the last chance for the work to prove it can carry its boundaries before becoming public and durable.
